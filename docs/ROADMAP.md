# redis-rs Toy Project Roadmap（面试展示版）

## 项目定位

把 `redis-rs` 定位为“能跑、能讲、能演示”的 toy project：
优先做 **够用**（核心功能闭环）、**可讲清楚**（架构简单明确）、**可演示**（5-8 分钟稳定 demo），避免过度工程化和无限扩展。

当前结构基线：`src/`（核心实现）、`tests/`（测试）、`perf/`（压测脚本）、`docs/`（文档）。

---

## 阶段 1（本周）：稳定可演示的最小闭环

### 目标
完成一个面试时可稳定演示的 Redis-like 核心闭环：协议、命令执行、基础数据结构、基础测试、基础文档。

### 交付物
- 在 `src/` 明确并冻结“演示白名单命令”（建议：`PING/SET/GET/DEL/EXPIRE` + 每类结构 1-2 个命令）。
- 在 `tests/common.rs` 基础上补齐最小回归用例（重点覆盖字符串、TTL、至少 2 类复合结构）。
- 在 `docs/` 新增：
  - `docs/ARCHITECTURE.md`（请求链路：RESP -> Command -> Store -> Persistence）
  - `docs/DEMO.md`（面试 demo 指令清单）
  - `docs/ROADMAP.md`（本文）

### 验收标准
- 本地可一键启动 `redis-server` 与 `redis-cli` 并完整走通 demo。
- 核心命令执行结果与预期一致（含过期行为）。
- 文档可让面试官在 5 分钟内理解项目边界与主流程。

### 预计投入
- 8-12 小时。

---

## 阶段 2（两周内）：可解释的工程质量

### 目标
在不显著增加复杂度的前提下，提高可信度和可讲述性：补测试覆盖、补性能基线、补设计取舍说明。

### 交付物
- 为 `src/commands/mod.rs`、`src/commands/data.rs` 补“命令分层”说明（输入校验、执行分发、错误语义）。
- 为 `src/store/` 增补边界测试（空 key、负索引、类型冲突、TTL 清理时机）。
- 使用 `perf/main.py` 产出 1 份基线结果（如 mixed/set/get），归档到 `perf/results/`。
- 在 `docs/` 新增 `docs/TRADEOFFS.md`（为什么只做 RESP2、为何暂不做事务等）。

### 验收标准
- 测试覆盖“演示路径 + 关键边界路径”。
- 有一份可复现性能数据（不追求绝对值，只追求可解释）。
- 能清楚回答“为什么这样设计，而不是更复杂方案”。

### 预计投入
- 15-25 小时。

---

## 阶段 3（1-2 月，可选）：有亮点但不失控的增强

### 目标
只增加 1-2 个面试亮点能力，而不是全面扩展。

### 交付物（2 选 1 或 2）
- 方向 A（持久化亮点）：围绕 `src/persistence/` 做可解释增强（如 AOF 重写触发策略可观测、启动恢复流程可视化）。
- 方向 B（观测性亮点）：增加轻量运行指标输出（请求数、命中率、过期淘汰次数）并补文档。
- 方向 C（演示体验）：整理一个“30 秒手工重放流程”，让 demo 更稳定、更少人工操作。

### 方向说明（面试可讲版）
- 持久化可解释性：重点不是“做很强的持久化”，而是能讲清楚数据何时落盘、重启如何恢复、故障边界在哪里。落地方式可包括在 `src/persistence/manager.rs` 输出触发与耗时信息，在 `src/persistence/aof.rs` 或 `src/persistence/rdb.rs` 输出恢复统计，并在 `docs/TRADEOFFS.md` 明确保证与不保证。
- Demo 重放流程：把面试演示流程整理成固定步骤，减少手敲失误并保证可复现。落地方式可包括 `docs/DEMO.md` 里的启动、连通性检查、命令序列和手工兜底流程。

### 验收标准
- 新增能力可在面试中 1 分钟解释价值、1 分钟演示效果。
- 不引入大规模重构，不破坏现有演示稳定性。
- 文档明确“能力边界 + 未覆盖场景”。

### 预计投入
- 20-40 小时（可随时止损）。

## Scope Guard（明确不做）

1. 不做 Redis 全命令兼容，不追求 production-ready。
2. 不做集群、主从复制、哨兵、分片。
3. 不做 Lua、事务（`MULTI/EXEC`）、发布订阅、ACL。
4. 不做复杂配置系统与平台化运维能力。
5. 不做“为了炫技”的微优化（先保证可解释与可演示）。
6. 不做重 CI/CD、混沌测试、超大规模压测体系。

---

## 与仓库结构的落地映射

- `src/`：只维护演示主链路相关实现与少量边界修复。
- `tests/`：优先集成级 smoke + 关键回归，不追求覆盖率数字。
- `perf/`：保留轻量基线测试，输出可对比结果即可。
- `docs/`：沉淀“能讲清楚”的文档（架构、取舍、demo、roadmap）。

## 面试展示脚本（5-8 分钟）

### 讲解大纲（时间分配）
1. **0:00-0:40** 项目定位：toy project，目标是“够用、可讲、可演示”。
2. **0:40-1:50** 架构主线：`src/resp.rs` 编解码 -> `src/commands/` 解析执行 -> `src/store/` 数据结构 -> `src/persistence/` 持久化。
3. **1:50-3:10** 关键设计取舍：只做 RESP2、命令白名单、并发模型与 TTL 处理。
4. **3:10-6:30** 实机 demo（见下方流程）。
5. **6:30-7:30** 复盘与边界：做了什么、没做什么、后续可选方向。

### Demo 流程（建议）
1. 启动服务端：`cargo run --bin redis-server`
2. 客户端连入：`cargo run --bin redis-cli`
3. 基础链路：`PING`、`SET k v`、`GET k`、`DEL k`
4. TTL 能力：`SET temp 1 EX 1`，等待后 `GET temp`
5. 结构化能力（任选 2 组）：
   - List：`RPUSH l a b` + `LINDEX l -1`
   - Hash：`HSET h f v` + `HGET h f`
   - ZSet：`ZADD z 1 m1 2 m2` + `ZCARD z`
6. 收尾：展示 `tests/common.rs` 与 `perf/main.py`，说明已具备基础回归与轻量压测。
